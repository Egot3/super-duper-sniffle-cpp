# Суть проекта

Это финальный проект курса C++. Вы напишете консольную библиотеку для работы с растровыми изображениями: будете читать и писать BMP, преобразовывать изображения и сохранять их в собственных двоичных форматах — несжатом и сжатом (с палитрой).

Готовая система должна уметь:

- читать и записывать изображения в собственном несжатом формате (далее `UncompressedImage`);
- читать и записывать изображения в собственном сжатом формате с палитрой (далее `CompressedImage`);
- читать BMP в `UncompressedImage` и сохранять `UncompressedImage` обратно в BMP (готовая библиотека для работы с BMP уже предоставлена);
- выполнять отражение, поворот, инверсию (негатив) и преобразование в оттенки серого;
- применять произвольный ядерный фильтр (kernel filter) к изображению;
- поверх ядерных фильтров реализовать повышение резкости, размытие и выделение краёв.

## Как считаются баллы

По мере корректной реализации функций и структур проекта ваш код будет проходить всё больше тестов. Каждый тест проверяет конкретный функционал и начисляет указанное в нём количество баллов (см. `tests/tests.cpp`, `TEST_AWARD_POINTS`).

Два первых теста — **обязательные** (compulsory): `BMP read` и `BMP save and read`. Если они не пройдены, итоговая сумма обнуляется, даже если другие тесты прошли. Логика этого описана в классе `TestAwarder` в `tests/tests.cpp` — посмотрите, как он работает, это полезно само по себе.

Чтобы зафиксировать баллы, отправьте решение в Яндекс.Контест. Все решения проверяются на плагиат.

Краткая сводка по тестам (детали и точные количества баллов — в `tests/tests.cpp`):

| Группа тестов | Баллов | Что проверяется |
| --- | ---: | --- |
| `BMP read`, `BMP save and read` | 1 + 1 | **Обязательно**. Загрузка BMP в `UncompressedImage` и round-trip BMP→BMP |
| `Simple/Horizontally/Vertically mirrored` | 2 + 1 + 2 + 1 | Отражения по обеим осям (шаблонный `mirror`) |
| `Simple grayscale` / `Grayscale conversion` | 3 + 2 | Перевод в оттенки серого |
| `Simple negative` / `Negative conversion` | 2 + 1 | Негатив |
| `Rotation` (8 кейсов: малое/большое изображение × прямые/произвольные углы × без/с интерполяцией) | 5 + 8 + 5 + 8 + 3 + 6 + 3 + 5 | Поворот на любой угол с заливкой и (опционально) интерполяцией |
| `Identity / Edge / Blur / Sharp kernel` | 2 + 4 + 4 + 4 | Применение ядерных фильтров |
| `Color distance` | 1 | Корректность `colorDistanceSq` |
| `Compressed image` (4 кейса) | 7 + 7 + 6 + 6 | Конвертация в `CompressedImage` и обратно, чтение и запись на диск |

Суммарно — **100 баллов**.

6. После подключения вы сможете работать в VS Code привычным образом: слева — проводник файлов WSL, снизу — консоль, в которой выполняются команды. Создание, редактирование и удаление файлов работает так же, как в обычном VS Code.

# Сборка и запуск

## Выбор компилятора

В начале `Makefile` указан компилятор:

```MakeFile
CXX := g++
```

Если хотите использовать `clang++`, замените `g++` на `clang++` в этой строке. Версия стандарта — `C++20` (тоже задана в `Makefile`).

## Цели Makefile

- `make build` — собрать проект (без тестов).
- `make run` — собрать и запустить `src/main.cpp`. Этим файлом удобно отлаживать свой код вручную.
- `make test` — собрать **и запустить** тесты. Тесты подключают весь `src/`, кроме `main.cpp`.
- `make clean` — удалить артефакты сборки (`build/`, `tmp_images/`, `logs/`).
- `make clean test` — два таргета через пробел: сначала `clean`, затем `test`. То есть полная пересборка тестов.
- `make build run` — аналогично: сборка, затем запуск. Удобно использовать, когда нестабильное состояние артефактов сборки.

## Первая проверка

Из корня проекта:

```Shell
make run   # должна быть успешная сборка и вывод Hello, World!
make test  # должна быть успешная сборка, провал тестов и 0 баллов
```

Если оба этапа отработали (с ожидаемыми провалами тестов) — окружение настроено.

## Предупреждения = ошибки

В `Makefile` стоит флаг `-Werror`, поэтому любое предупреждение компилятора прерывает сборку. **В Яндекс.Контесте настроены те же флаги.** Будьте внимательны: не оставляйте `unused`-переменных, неинициализированных полей, неявных приведений и т.п.

## Стиль кода

В корне лежит `.clang-format` — конфигурация автоформатирования (на основе Google-стиля). Включите автоформат на сохранение или вызывайте его вручную (`clang-format -i <file>`). Это сильно облегчает чтение кода и снижает количество замечаний при проверке.

# Основы C++: заголовочные и исходные файлы

Самое главное, что появляется в этом проекте по сравнению с предыдущими — теперь код размещается не только в `main.cpp` и не только в файлах `.cpp`, но и в `.h` (header / заголовочных) файлах.

**Заголовочные файлы (`.h`)** содержат *объявления* функций, классов, структур и глобальных переменных. Их подключают через `#include` другие файлы, чтобы видеть эти объявления. Например, когда вы пишете `#include <iostream>`, препроцессор подставляет на место этой строки содержимое файла `iostream`, в котором объявлены `std::cout`, `std::cin`, `std::ostream`, `std::istream` и т.д. — но *не реализованы*.

**Исходные файлы (`.cpp`, source files)** содержат *реализацию* функций и методов классов. Один заголовочный файл, как правило, имеет «парный» исходный с тем же именем.

Пример: программа, которая считывает точки в вектор, сортирует его и выводит.

`include/point_utils.h`:

```C++
#pragma once

#include <cstdint>
#include <vector>

struct Point {
    int64_t x = 0;
    int64_t y = 0;
    int64_t z = 0;
};

void inputPoints(size_t n, std::vector<Point>& points);

void solve();
```

`src/point_utils.cpp`:

```C++
#include <iostream>

#include "point_utils.h"

void inputPoints(size_t n, std::vector<Point>& points) {
    points.resize(n);
    for (size_t i = 0; i < n; ++i) {
        std::cin >> points[i].x >> points[i].y >> points[i].z;
    }
}

void solve() {
    size_t num;
    std::cin >> num;
    std::vector<Point> points;
    inputPoints(num, points);
    std::cout << "Entered " << num << " points\n";
}
```

Обратите внимание на правила `#include`:

- в каждом файле подключайте именно то, что в нём используется — не больше и не меньше;
- `<iostream>` подключается только в `point_utils.cpp`, потому что нужен только там;
- `<vector>` достаточно подключить один раз — в `point_utils.h`, ведь любой файл, подключающий этот заголовок, через него получит и `<vector>` тоже;
- в заголовочные файлы добавляйте `#pragma once` (или классические include guards), чтобы избежать повторного включения.

Чтобы скомпилировать такую программу вручную, нужна команда вида:

```Shell
g++ -I<include_directory> <source_file_1> <source_file_2> -o <output_executable>
```

Для нашего примера: `g++ -Iinclude src/point_utils.cpp -o main` соберёт исполняемый файл `main`, который можно запустить как `./main`. В этом проекте всё то же самое делает за вас `Makefile`.

# Работа с библиотекой BMP

Готовая библиотека `libbmp` поддерживает несжатые 24/32-битные BMP — этого достаточно для всех тестовых картинок в `images/`. Сигнатуры доступных методов смотрите в `include/libbmp.h` (это короткий и понятный файл — обязательно прочитайте его перед использованием).

Понять конкретное поведение библиотеки можно так:

- из примеров ниже;
- скопировав примеры в `main.cpp` и запустив `make run`;
- прочитав `include/libbmp.h` и `src/libbmp.cpp`.

Создание и запись BMP-файла:

```C++
BMP bmp(2, 2);
bmp.set_pixel(0, 0, 255, 0, 0);
bmp.set_pixel(1, 0, 255, 255, 0);
bmp.set_pixel(0, 1, 255, 0, 255);
bmp.set_pixel(1, 1, 0, 255, 0);
const std::string filename = "image.bmp";
bmp.write(filename.c_str());
```

Чтение BMP-файла:

```C++
const std::string filename = "image.bmp";
BMP bmp(filename.c_str());
bmp.get_width();
bmp.get_height();
uint8_t r, g, b;
bmp.get_pixel(0, 0, r, g, b);
// теперь r, g, b содержат значения каналов RGB у пикселя в точке (0; 0)
```

# Форматы собственных файлов изображений

Для записи и чтения собственных форматов `UncompressedImage` и `CompressedImage` используются двоичные файлы со следующей структурой. Все многобайтовые числа пишутся в **little-endian** (это естественный порядок байт на x86/x64), без выравнивания.

## Несжатый формат (`RAWIMAGE`)

```
====RAW FILE FORMAT====
- 8 байт:  сигнатура формата. Должна быть равна строке "RAWIMAGE" (ровно 8 ASCII-символов, без кавычек и завершающего нуля)
- 3 байта: версия формата. Три байта подряд: 1, 0, 0
- 4 байта: width  (uint32_t)
- 4 байта: height (uint32_t)
- 1 байт:  is_grayscale (1 — grayscale, 0 — RGB)
ЕСЛИ is_grayscale == 1:
- width * height байт: значения серого пикселя (по 1 байту на пиксель), порядок — построчно сверху вниз, в строке — слева направо
ИНАЧЕ:
- width * height * 3 байт: RGB-пиксели (тройки байт r, g, b в том же порядке)
- 9 байт: сигнатура конца файла. Должна быть равна строке "RAWIMGEND" (ровно 9 ASCII-символов)
```

## Сжатый формат (`CMPRIMAGE`)

```
====COMPRESSED FILE FORMAT====
- 10 байт: сигнатура формата. Первые 9 байт — строка "CMPRIMAGE", 10-й байт — 0x00 (нулевой)
- 3 байта: версия формата. Три байта подряд: 6, 6, 6
- 4 байта: width  (uint32_t)
- 4 байта: height (uint32_t)
- 1 байт:  pow — логарифм по основанию 2 размера палитры (т.е. палитра содержит 2^pow цветов)
- 2^pow * 3 байт: палитра (тройки байт r, g, b для каждого цвета, индексируются по порядку: 0, 1, 2, ...)
- width * height байт: id цвета из палитры для каждого пикселя (тот же порядок, что и в RAW)
- 10 байт: сигнатура конца файла. Должна быть равна строке "CMPRIMGEND" (ровно 10 ASCII-символов)
```

> Подсказка: байтовые длины строк имеет смысл проверять буквально — `sizeof("RAWIMAGE") - 1 == 8`, потому что строковый литерал в C++ содержит завершающий `\0`, который в формат файла **не** записывается. Точно так же при чтении не забывайте, что `std::string`, инициализированная как `std::string s(8, '\0')`, выделит 8 байт и в неё нужно читать ровно 8 байт.
